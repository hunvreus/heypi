# Prompt design analysis

Status: implemented after review by Claude and Grok. This document retains the previous prompt
shape as rationale for the resulting design.

## Scope

This note describes the model input generated by HeyPi for a normal chat turn, compares it with the local Eve, pi-chat, and Hermes Agent snapshots, and proposes a smaller Pi-native design.

The comparison was made on 2026-07-20. It covers prompt construction, not chat UI such as Slack status, reactions, or todo rendering.

## Conclusion

Before this change, HeyPi supplied the necessary pieces but placed too much tool documentation in the system prompt and too little normalized chat identity there. The largest avoidable cost was the generated `Available tools` section: every active tool's full description was repeated in the system prompt even though the model already received the same description and parameter schema as a tool definition.

The target should be:

1. Keep the system prompt stable and short.
2. Put tool-specific behavior in each tool definition.
3. Send HeyPi-asserted actor and conversation metadata with each user turn.
4. List skill metadata in the system prompt, but load skill bodies only when relevant.
5. Materialize skills inside the runtime so reads, scripts, and assets use one path and one security boundary.
6. Keep approvals in the execution layer. They are runtime state, not model instructions.

## Previous HeyPi input

### System prompt

For Codex Tag, Pi previously assembled these blocks in order:

1. `agent/system.md`
2. `agent/instructions.md`
3. HeyPi's chat and filesystem suffix
4. Pi's available-skills index
5. Pi's current date and working directory
6. HeyPi's generated `Available tools` section

The authored portion is:

```text
You are Codex Tag, a Pi-powered coding agent reachable from team chat.

Keep chat updates concise. Use todo updates for substantial multi-step work. Do not dump large files
into replies.

Do not inspect the repository for greetings, vague offers, or incomplete coding requests. Ask what
the user wants built or changed.

When working in a repository, inspect first, then act. Prefer focused searches and targeted file
reads. Install dependencies when required to validate a project, but avoid unnecessary setup.

Default workflow:

- If the user has not given a concrete coding goal, ask a short clarifying question before using tools.
- Understand the request before editing.
- Use the current workspace as the source of truth.
- Prefer the local checkout and GitHub CLI over web search for repository, issue, and pull-request work.
- For explicit pull-request requests, work in a dedicated branch and worktree, then commit, push, and open the pull request.
- Run focused checks before reporting success.
- Store durable repo or workflow facts in memory when they will help future turns.
- Use chat_history only when the current message lacks needed context.
```

HeyPi then appends:

```text
Incoming chat messages are supplied as the current chat delta. Reply in the same remote thread.

Use /workspace for this channel or DM. Use /shared only for reusable adapter-level files. Do not put secrets or private channel-specific content in /shared.

Use staged agent skills, tools, and extensions when they apply.
```

Pi adds a skill index containing each skill's name, description, and rewritten `/agent/...` path, followed by the current date and `Current working directory: /workspace`.

Finally, HeyPi appends all active tools. Codex Tag normally exposes:

- Runtime files and commands: `read`, `bash`, `edit`, `write`, `grep`, `find`, `ls`
- Chat: `chat_history`, `chat_attach`, `chat_request_secret`
- State: `todo`, `memory`, `memory_search`
- Agent-defined tools: `fetch_page`, `web_search`

Each entry repeats the tool description and then any `promptGuidelines`. The tool API separately sends the same description plus its JSON parameter schema. The repetition exists because Pi's custom-system-prompt branch includes skills, date, and working directory but omits tool snippets and guidelines.

### User turn

HeyPi sends only the current chat delta. Earlier conversation is retained in the Pi session; older remote messages not present in that session are available through `chat_history`.

A representative Slack root mention is:

```text
<chat_context>
{"adapter":"slack","adapterId":"slack","surface":"thread","trigger":"mention","conversation":"C123","session":"1710000000.000100","thread":"1710000000.000100","message":"1710000000.000100","actor":{"id":"U123","bot":false},"mentioned":true,"time":"2026-07-20T16:00:00.000Z"}
</chat_context>
<chat_messages>
- [uid:U123] unknown: Please fix issue 184 and open a PR.
</chat_messages>
```

A representative follow-up in that Slack thread is:

```text
<chat_context>
{"adapter":"slack","adapterId":"slack","surface":"thread","trigger":"message","conversation":"C123","session":"1710000000.000100","thread":"1710000000.000100","message":"1710000030.000200","actor":{"id":"U123","bot":false},"mentioned":false,"time":"2026-07-20T16:00:30.000Z"}
</chat_context>
<chat_messages>
- [uid:U123] unknown: Also update the changelog.
</chat_messages>
```

The envelope is useful and machine-readable, but currently has four defects:

- Slack usually provides no actor display name on the event object used by HeyPi, so the model often sees `unknown`.
- It contains implementation fields the model does not need, including `session`, `message`, `mentioned`, and `bot: false`.
- It usually lacks a human-readable channel name.
- A Slack thread follow-up without an explicit reply reference is labeled `message`, even though its semantic trigger is a thread continuation.

The static instruction to reply in the same remote thread is also too Slack-shaped. HeyPi routes delivery, so the model only needs to know it is replying to the active remote conversation.

### Conditional context

Other input is added only when applicable:

- Recalled memory is inserted in a `<memory-context>` block immediately before the current user message and explicitly marked as untrusted reference material.
- A skill body enters context after the model reads its advertised file.
- Custom tools and extensions add tool definitions and may add prompt guidelines.
- Pi retains prior messages and compaction summaries in the session.
- Todo reconciliation can inject a hidden follow-up if the model tries to finish with unresolved items.

Approvals do not alter the prompt. HeyPi intercepts a protected tool call, parks the turn, obtains a decision, and resumes or rejects that same call. This is the correct boundary.

### Skills and runtime mismatch

Agent resources are staged under HeyPi's host state directory. The prompt rewrites their locations to `/agent`, but `/agent` is not currently mounted in the runtime. A special read broker maps `read /agent/...` back to the staged host file. Other runtime tools do not share that view, so a skill's scripts and assets are not generally executable or readable through normal runtime commands.

This split is conceptually inconsistent. A skill package should have one model-visible location with the same contents available to `read`, `bash`, and any other runtime file tool while protecting canonical staged content from runtime changes.

## Comparison

| Concern | HeyPi | Eve | pi-chat | Hermes Agent |
| --- | --- | --- | --- | --- |
| Base prompt | Authored replacement plus generated suffix | Small authored composition | Pi default prompt plus a large chat suffix | Large layered identity/session prompt |
| Tool guidance | Full descriptions and guidelines duplicated into system | Tool definitions carry most behavior; one short parallelism rule is in system | Pi default tool prompt plus custom tool descriptions | Tool schemas carry most behavior; conditional model guidance is added to system |
| Chat context | Structured JSON per turn across adapters | Adapter-specific context block per turn; Slack includes actor, channel, and thread IDs | Channel service, mode, and name in system; transcript carries UID and name | Platform formatting hint in system; gateway/session supplies message context |
| Identity | Actor ID is present; display name is adapter-dependent | Slack context includes verified ID and resolved names when available | Explicitly says UID is authoritative and names are spoofable | Platform-oriented, with less explicit identity guidance in the base prompt |
| Skills | Pi skill index; body loaded with `read`; `/agent` is host-brokered | Compact index plus dedicated `load_skill`; complete packages live in runtime workspace | Complete skill packages live under runtime `/workspace/skills` or `/shared/skills`; body loaded with `read` | Verbose mandatory index plus `skill_view`; linked files and scripts can be inspected |
| Todo | Detailed `promptGuidelines`, visible adapter projection, final reconciliation | Detailed tool contract and durable state | No equivalent built-in todo in the inspected snapshot | Detailed tool schema and todo restoration after history/compression |
| Approvals | Execution-layer policy that parks and resumes tool calls | `needsApproval` execution policy and durable pending input | No comparable user approval flow in the inspected snapshot | Dangerous-command approval is execution/UI policy, not core system prose |
| Memory | Curated tools plus scoped recall block | Durable runtime/session facilities | Files in `/workspace` and `/shared`, explained in the system suffix | Persistent memory and profile may be frozen into the system prompt |
| Prompt stability | Mostly stable, but generated from active tools and contains duplicated prose | Deliberately stable for session caching | Chat suffix is conditionally appended on dispatch | Cached once per session and rebuilt after compression |
| Overall style | Structured but redundant | Smallest and cleanest separation | Direct and understandable, but hardcoded and chat-specific | Most capable and most verbose |

### Eve

Eve is the best architectural reference for separation of concerns. Its base composition is authored instructions, optional workspace information, one parallel-tool rule, connections, and a compact skill index. Tool behavior remains in tool definitions. Slack actor, channel, and thread metadata is delivered as a separate `<slack_context>` entry rather than frozen into the system prompt. Approvals remain harness state.

Its dedicated `load_skill` tool gives a clear activation event and keeps skill bodies out of the stable prefix. That is useful, but HeyPi does not need to add this API merely for parity because Pi already has a conventional skill index and `read` workflow.

### pi-chat

pi-chat is the closest behavioral reference because it also wraps Pi for remote coding work. It explicitly names the service, mode, and channel; explains delta semantics; marks UID as authoritative; and mounts complete skill packages into the runtime. These are good ideas.

Its drawback is concentration: chat behavior, Alpine details, memory, environment restoration, skills, attachments, and history are all hardcoded in one long suffix. That works for one chat worker model but is less suitable for HeyPi's adapter and runtime matrix.

### Hermes Agent

Hermes demonstrates two useful principles: cache the assembled system prompt for the session, and keep detailed todo behavior in the static tool schema. It also has dedicated skill tools that can load related files.

It is not a minimal prompt reference. Identity, memory, provider workarounds, model-specific enforcement, skills policy, context files, environment hints, timestamp, and platform formatting can all enter the system prompt. Its mandatory skill prose is especially aggressive and should not be copied.

## Proposed design

### Ownership rules

Use the following placement rules:

| Information | Owner |
| --- | --- |
| Agent identity and domain workflow | Agent `system.md` and `instructions.md` |
| Stable chat and filesystem contract | HeyPi system suffix |
| Tool purpose, parameters, and usage rules | Tool definition |
| Available skill name, description, and path | Pi skill index |
| Skill instructions, scripts, and assets | Read-only runtime skill mount |
| Actor, adapter, conversation, trigger, and timestamp | Current user turn |
| Recalled memory and remote history | Explicit untrusted context/tool results |
| Approval policy, pending state, and decision | Execution layer and tool result |
| Todo state | Todo controller, tool result, and compaction/session restoration |

This removes the need for a general prompt builder that knows the semantics of every tool or adapter.

### Minimal system suffix

Keep the authored agent prompt unchanged, then append only:

```text
## Chat runtime

You are responding through HeyPi. Each turn includes HeyPi-asserted context followed by user-authored messages. actor.id is the authoritative identity value; names are display labels. HeyPi, not you, enforces access and tool permissions. HeyPi routes your final response to the active conversation.

Use /workspace for conversation files. Use /shared only for reusable adapter-level files; never store secrets or conversation-private content there. Agent skills are managed under /agent/skills. Do not modify them. When a listed skill clearly applies, read it before acting.
```

Omit `/shared` or `/agent/skills` sentences when those facilities are unavailable. Scheduled runs should use a separate one-sentence suffix and no chat contract.

Pi can continue to append its compact skill index, current date, and `/workspace` working directory.

Do not append full tool descriptions. Either:

- Preferred: move each tool's complete behavioral rules into its API description, as Hermes does for todo.
- Transitional: append one generated `Tool guidance` list containing only non-empty `promptGuidelines`, deduplicated in active-tool order.

The transitional form preserves Pi's intended semantics while removing the duplicated descriptions. Tools without special guidelines add no prompt text.

### Minimal turn envelope

Normalize adapter events into one small HeyPi-asserted shape:

```text
<chat_context>
{"adapter":"slack","account":"slack","conversation":{"id":"C123","kind":"thread","name":"basecoat","thread":"1710000000.000100"},"actor":{"id":"U123","name":"Ronan Berder"},"trigger":"reply","time":"2026-07-20T16:00:30.000Z"}
</chat_context>
<chat_messages>
- [uid:U123] Ronan Berder: Also update the changelog.
</chat_messages>
```

Required fields should be `adapter`, `conversation.id`, `conversation.kind`, `actor.id`, `trigger`, and `time`. Include `account`, names, groups, and thread/topic identifiers only when available or relevant. Do not send internal session keys, duplicate booleans, or message IDs unless a tool needs them.

Adapters should resolve display names when inexpensive, but IDs remain authoritative. They should also normalize continuation semantics before prompt construction: a message continuing an active Slack thread, Discord reply chain, or Telegram reply chain should be `reply`, regardless of the source API's raw field names.

### Skills

Materialize the complete staged skill tree at `/agent/skills` inside every runtime:

- Read-only mounts where supported; otherwise disposable copies refreshed from canonical staged content
- Directory traversal allowed only inside that tree
- Scripts executable when their file mode allows it
- No host path exposed to the model
- Same path for `read`, `bash`, and custom runtime tools

Then remove the special host-side `/agent` read broker. Agent tools or libraries that must also be available at runtime should use explicit sibling mounts such as `/agent/tools` only when there is a concrete consumer; do not mount the entire agent source by default.

Continue using Pi's name/description/path skill index and on-demand `read`. A dedicated `load_skill` tool would provide nicer telemetry and stricter access, but it is additional machinery rather than a requirement.

### Custom tools

Custom tools should follow the same contract as built-ins:

- The description says when and why to call the tool.
- The schema describes each input precisely.
- `promptGuidelines` are reserved for short behavioral invariants that Pi must see outside the schema.
- Tool registration order is deterministic so the cached prefix is stable.
- Tool availability is authoritative; the system prompt should not maintain a second hand-written list.

### Approvals

No approval instructions are needed in the static system prompt. The model calls a tool normally. HeyPi decides whether approval is required, pauses the exact call, and returns an approved, rejected, canceled, or expired result.

Two invariants matter:

- A pending call is idempotent and cannot be executed twice after duplicate resume events.
- A rejection result clearly states that the action did not run, so the model can choose a safe alternative or report the block.

This works for built-in and custom tools without teaching the model adapter-specific approval UI.

### Todo and memory

Todo's detailed rules belong in the tool definition. Keep only the agent-level preference, such as "Use todo updates for substantial multi-step work," in authored instructions. Visible Slack/Discord/Telegram todo rendering is an adapter projection of controller state and should not affect prompting.

Memory recall should remain a clearly delimited, untrusted context block near the current user turn. Memory-writing policy belongs in the memory tool definition. Do not freeze changing memory into the system prompt because that harms cache stability and makes provenance less clear.

### Expected result

The model receives four understandable layers:

1. Who it is and how this agent works
2. A short, stable HeyPi runtime contract
3. Skill metadata and tool schemas
4. Trusted current-chat metadata plus user-authored content

This is smaller than the current HeyPi prompt, less hardcoded than pi-chat, substantially less verbose than Hermes, and preserves Eve's clean separation without adding a new skill API.

## Recommended implementation order

1. Stop duplicating active tool descriptions; retain only deduplicated guidelines.
2. Compact the chat envelope and fix adapter-level name and reply normalization.
3. Replace "same remote thread" with adapter-neutral delivery language.
4. Expose managed `/agent/skills` in every runtime and remove the host read broker.
5. Move remaining HeyPi custom-tool guidelines into tool descriptions where they describe only that tool.
6. Add one prompt snapshot test for the stable system shape and table-driven tests for normalized Slack, Discord, and Telegram turn envelopes.

Do not add separate snapshot tests for every tool combination. Contract tests for tool registration plus one representative prompt are sufficient.
